# L — 统一用户资产 + 存储抽象 + 生图鉴权 + 文件库在线服务化

状态：设计定稿，进入实施。本文是实施级契约。

## 背景（审计结论）

- **存储路径硬编码且不一致**：`api/images.py:32` 用 `parents[3]`→`rpg/` 上一级；`frontend_routes.py`/`library.py`/`script_import.py`/`runtime.py` 用 `parents[1]`/`parent.parent`→`rpg/platform_data/`；`app.py` 的 `uploads/` 在 `platform_data/` 外。**无集中根常量**。生产上 ai_images 与 avatars 物理分两处。
- **文件库**：`assets` 表(id,user_id,name,rel_path,mime,kind,size) + `library.py`/`api/library.py`，因"无 MIME 白名单可任意上传"被 `RPG_LIBRARY_ENABLED` 禁用。无 source/关联字段。
- **导入 txt**：存 `platform_data/scripts/user_{id}/`，`scripts.source_path` 引用，**未登记资产表**。
- **生图鉴权漏洞**：`POST /api/images/generate` + `generate-persona-image` 入队前不校验 attach 目标归属(worker 侧才校验→耗配额/信息泄露);`upsert_user_card` 接受 body 直传 `avatar_path`(URL 注入)。worker 侧 attach 归属校验到位。
- **缺手动上传**：仅用户头像有手动上传;卡头像/剧本封面/人设图只有 AI 生图。

## 北星

一个**统一用户资产子系统**：所有用户文件(AI 生图/头像/导入 txt/卡图)登记到单一 `user_assets` 表，物理存储经**单一集中存储模块**(`platform_app/storage.py`,根可配置 env、OSS 可插拔、去掉所有 `parents[N]` 硬编码)。文件库 UI = 只读管理(列表/查看/下载/删除),**不支持手动上传**(内容只来自功能组件)。删除带关联检查(引用了卡/剧本/头像则警告会显示错误,无引用则二次确认)。每个 AI 生图位置都配**手动上传/更换**,生成/更换严格按编辑权限归属鉴权。

## S1 — 存储抽象 `platform_app/storage.py`（基座，去硬编码）

**单一真相源**：
```python
PLATFORM_DATA_ROOT = Path(os.environ.get("RPG_DATA_ROOT",
    str(Path(__file__).resolve().parents[1] / "platform_data")))  # 默认 rpg/platform_data
AI_IMAGES_DIR = PLATFORM_DATA_ROOT / "ai_images"
AVATARS_DIR   = PLATFORM_DATA_ROOT / "avatars"
SCRIPTS_DIR   = PLATFORM_DATA_ROOT / "scripts"
LIBRARY_DIR   = PLATFORM_DATA_ROOT / "library"
UPLOAD_CHUNKS_DIR = PLATFORM_DATA_ROOT / "upload_chunks"
```
- `store_bytes(data, *, kind, filename) -> (storage_key, url)`：落盘(mkdir -p)，返回 storage_key(=`{kind}/{filename}`) + 相对 URL。**OSS 替换点=此函数体**。
- `resolve_path(storage_key) -> Path`：服务/删除用，做路径穿越防护(必须在 PLATFORM_DATA_ROOT 下)。
- `delete_file(storage_key)`：删物理文件(missing_ok)。
- `public_url(storage_key) -> str`：物理键→对外 URL。OSS 时返 CDN URL。
- **统一服务路由** `GET /api/storage/{kind}/{name}`(白名单 kind + 扩展名 + 穿越防护 → FileResponse)。**保留旧路由** `/api/images/file/*`、`/api/profile/avatar/file/*` 作别名(老 URL 不破)。
- **改造**：`api/images.py` 改用 storage(`store_image`/serve 走 storage,去掉 `parents[3]`);`frontend_routes.py` 头像、`script_import.py` txt 落盘也改走 storage(统一根)。
- **生产数据迁移**(部署步骤,非代码迁移):把 prod 现有 `/opt/rpg-roleplay/platform_data/ai_images/*`(parents[3] 旧位置)`mv` 到统一根 `/opt/rpg-roleplay/rpg/platform_data/ai_images/`(URL 不变,DB 不动,只挪物理文件)。

## S2 — 统一资产表 `user_assets`（migration v72）

```sql
create table user_assets(
  id bigint generated by default as identity primary key,
  user_id bigint not null references users(id) on delete cascade,
  kind text not null,            -- 'ai_image'|'avatar'|'script_txt'|'card_image'|'cover'
  storage_key text not null,     -- storage 模块的唯一键(kind/filename)
  url text not null default '',  -- 对外 URL
  source text not null default '', -- 'image_gen'|'avatar_upload'|'script_import'|'manual_upload'
  ref_kind text,                 -- 关联目标 'card'|'script'|'user'|'persona'
  ref_id bigint,                 -- 关联目标 id
  mime text not null default '', size bigint not null default 0,
  meta jsonb not null default '{}'::jsonb,  -- prompt/model / txt encoding/chapters
  created_at timestamptz not null default now()
);
create index idx_user_assets_user_kind on user_assets(user_id, kind, created_at desc);
create index idx_user_assets_ref on user_assets(ref_kind, ref_id) where ref_id is not null;
```
- **登记点**(功能组件产生内容时写一行)：`image_jobs` 生图完成 `store_image` 后登记(kind=ai_image,source=image_gen,ref=attach 目标);`frontend_routes` 头像上传登记;`script_import` finish/import txt 登记(kind=script_txt,source=script_import,ref=script)。
- **backfill**(v72 内或单独脚本)：把现有 `ai_images`(status=done) + `users.avatar_url` + `scripts.source_path` 回填 user_assets(尽力而为,无则跳过)。
- 现有 `ai_images`/`card_persona_images` 表保留(异步态/历史用);user_assets 是面向"文件库"的统一登记。`assets` 旧表(library 手动上传用)退役/不再写。

## S3 — 生图鉴权修复

- `POST /api/images/generate` **入队前**校验 attach 归属:card_avatar→`character_cards WHERE id AND user_id`;script_cover→`scripts WHERE id AND owner_id`;不属于→403。`persona_image` 同理。
- `POST /api/me/character-cards/{id}/generate-persona-image` 入队前校验 card 属 user(404)。
- `upsert_user_card`/`upsert_persona` 的 body `avatar_path` 加**前缀白名单**(只允许空 / `/api/storage/` / `/api/images/file/` / `/api/profile/avatar/file/`),拒外部 URL 注入。
- 编辑权限统一:复用现有 owner 校验(user_cards owner、scripts owner_id、script_edit `_require_owner`)。公开卡/订阅剧本非 owner 不能改头像/封面(现有校验已挡,确认即可)。

## S4 — 每个 AI 生图位置配手动上传/更换（按归属鉴权）

- 后端新增 multipart 上传端点(都 require_user + ownership + MIME 魔数白名单 png/jpg/webp + 大小限):
  - `POST /api/me/character-cards/{id}/avatar`(校验 card 属 user → store → 写 avatar_path + user_assets)
  - `POST /api/scripts/{id}/cover`(校验 owner_id → store → 写 cover_image_url + user_assets)
  - `POST /api/me/character-cards/{id}/persona-images/upload`(校验 card 属 user → store → 插 card_persona_images + 翻 is_current + 写 avatar_path + user_assets)
  - 用户头像 `POST /api/profile/avatar` 已有,补登记 user_assets。
- 前端:在每个生图位置(`GenerateImageModal` 旁 / 卡详情 / 剧本详情 / 人设图 tab)加"上传图片"按钮(file input → 上述端点)。可把 `GenerateImageModal` 升级成"AI 生成 / 上传图片"双 tab。非 owner 不显示这些操作(按编辑权限)。

## S5 — 文件库在线服务化（只读管理，不手动上传）

- **后端**(改造 `library.py`/`api/library.py`):
  - 删除/禁用 upload/mkdir(不支持手动上传)。
  - `GET /api/library`(从 `user_assets` 列该 user 全部资产,可按 kind 过滤,带 url/source/ref/size/created_at)。
  - `GET /api/library/asset/{id}`(查看单个,owner 校验)。
  - 下载走统一 `/api/storage/...`(或 `GET /api/library/asset/{id}/download` Content-Disposition)。
  - `POST /api/library/asset/{id}/delete`:先 `find_references(storage_key/url)` 查引用(卡/剧本/头像);返回引用列表给前端。**有引用**→前端弹窗警告"删除后这些卡/剧本/头像会显示错误",二次确认后:置空引用字段(avatar_path/cover_image_url/avatar_url=NULL,card_persona_images 删行) + `storage.delete_file` + 删 user_assets 行;**无引用**→单纯二次确认即删。
  - `RPG_LIBRARY_ENABLED` 闸移除(只读安全,无任意上传风险,默认开)。
- **前端**:文件库页(列表/网格,按 kind 分类,缩略图复用 AvatarImg,查看大图 lightbox,下载,删除弹窗带关联警告)。入口在用户菜单/设置。

## 关联反查（删除检查核心）

`storage.find_references(url) -> list`:查 `users.avatar_url` / `scripts.cover_image_url` / `character_cards.avatar_path` / `card_persona_images.image_url` 等值匹配(用户量有限,全扫可接受;大时给这几列加索引)。迁移到 user_assets 后用 `ref_kind/ref_id` 结构化反查。
